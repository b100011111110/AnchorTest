# LedgerBank API Design v1.0 — Full Detailed Specification

## 1. Introduction

This document defines the complete, deterministic, machine-parseable specification for a basic banking system covering account lifecycle and monetary operations. The intent of this document is to eliminate ambiguity and ensure that every aspect of the API—from input validation to persistence verification—is explicitly defined. This allows downstream systems such as automated test generators, constraint engines, and graph-based reasoning systems to operate without inference or assumptions.

The system is designed around five core operations:

1. Create Account
2. Delete Account
3. Get Account Details
4. Deposit Funds
5. Withdraw Funds

Each operation is described using strict typing, validation rules, execution ordering, concurrency behavior, and persistence guarantees.

---

## 2. Global Architectural Principles

### 2.1 Deterministic Behavior

Every endpoint must behave deterministically given the same inputs and system state. Non-deterministic outcomes are explicitly disallowed.

### 2.2 Strong Typing

All fields must map to primitive base types:

* Int
* Float
* String
* Bool
* Date (ISO8601)

Composite or nested objects are decomposed into explicitly defined primitive fields.

### 2.3 Explicit Validation

No implicit validation exists. Every validation rule must be declared including:

* Range checks
* Regex constraints
* Cross-field dependencies

### 2.4 State Transition Visibility

Each operation must define:

* Pre-condition (required system state)
* Post-condition (exact state mutation)

## 4. Endpoint: Create Account

### 4.1 Identity

* Method: POST
* Path: /v1/account
* Feature: Account_Management
* Idempotency: Non-Idempotent

### 4.2 Description

This endpoint provisions a new bank account. It initializes the account state, assigns a unique identifier, and seeds the account with an initial deposit. This operation is non-idempotent because repeated requests will create multiple accounts.

### 4.3 Request Fields

Field: name

* Type: String
* Required: True
* Min_Length: 3
* Max_Length: 100
* Regex: ^[A-Za-z ]+$
* Null: False

Field: dob

* Type: Date
* Required: True
* Min: 1900-01-01
* Max: Current Date
* Null: False

Field: initialDeposit

* Type: Float
* Required: True
* Min: 0
* Max: 10000000
* Null: False

### 4.4 Cross Field Constraints

IF initialDeposit < 1000 THEN reject request

### 4.5 Pre-Conditions

* User must be authenticated
* System must be operational

### 4.6 Validation Priority

1. Authentication
2. JSON schema validation
3. Field constraints
4. Business logic

### 4.7 Post-Conditions

* New row inserted into accounts table
* status = ACTIVE
* balance = initialDeposit
* interest_rate = 0.04

### 4.8 Success Response Schema

Field: account_id

* Type: String
* Regex: ^ACC[0-9]{6}$

Field: status

* Enum: [ACTIVE]

Field: balance

* Type: Float

Field: created_at

* Type: Date

### 4.9 Error Mapping

* AUTH_FAILED → 401
* INVALID_SCHEMA → 400
* INITIAL_DEPOSIT_TOO_LOW → 422

### 4.10 Persistence Verification

SELECT * FROM accounts WHERE account_id = ?

### 4.11 Concurrency

* No lock required
* Safe for parallel execution

## 5. Endpoint: Delete Account

### 5.1 Identity

* Method: DELETE
* Path: /v1/account/{id}
* Feature: Account_Management
* Idempotency: Idempotent

### 5.2 Description

Marks an account as deleted. Does not physically remove the record. Repeated calls yield the same result.

### 5.3 Path Parameter

Field: id

* Type: String
* Required: True
* Regex: ^ACC[0-9]{6}$

### 5.4 Pre-Conditions

* Account must exist
* status == ACTIVE

### 5.5 Post-Conditions

* status = DELETED

### 5.6 Validation Priority

1. Authentication
2. Path validation
3. Existence check
4. State check

### 5.7 Success Response

* account_id
* status

### 5.8 Error Mapping

* ACCOUNT_NOT_FOUND → 404
* ACCOUNT_NOT_ACTIVE → 409

### 5.9 Persistence Check

SELECT status FROM accounts WHERE id=?

### 5.10 Concurrency

* Row-level lock required

## 6. Endpoint: Get Account Details

### 6.1 Identity

* Method: GET
* Path: /v1/account/{id}
* Feature: Monitoring
* Idempotency: Idempotent

### 6.2 Description

Fetches the full state of an account. This is a read-only operation.

### 6.3 Pre-Conditions

### 6.4 Post-Conditions

### 6.5 Success Response Fields

* account_id
* name
* dob
* status
* balance
* interest_rate
* created_at

### 6.6 Error Mapping

* ACCOUNT_NOT_FOUND → 404

### 6.7 Concurrency

* Fully parallelizable

---

## 7. Endpoint: Deposit

### 7.1 Identity

* Method: POST
* Path: /v1/deposit
* Feature: Transaction_Operations
* Idempotency: Non-Idempotent

### 7.2 Description

Adds funds to an account. Creates a transaction record and updates balance.

### 7.3 Request Fields

Field: account_id

* Type: String
* Regex: ^ACC[0-9]{6}$

Field: amount

* Type: Float
* Min: 1
* Max: 10000000

### 7.4 Pre-Conditions

* Account exists
* status == ACTIVE
* amount > 0

### 7.5 Post-Conditions

* balance = balance + amount
* Insert transaction record

### 7.6 Error Mapping

* NEGATIVE_AMOUNT → 422
* ACCOUNT_NOT_FOUND → 404

### 7.7 Concurrency

* Row-level lock
* Optimistic concurrency supported

---

## 8. Endpoint: Withdraw

### 8.1 Identity

* Method: POST
* Path: /v1/withdraw
* Feature: Transaction_Operations
* Idempotency: Non-Idempotent

### 8.2 Description

Removes funds from an account. Ensures sufficient balance.

### 8.3 Request Fields

Field: account_id

* Type: String

Field: amount

* Type: Float
* Min: 1
* Max: 10000000

### 8.4 Pre-Conditions

* Account exists
* status == ACTIVE
* balance >= amount

### 8.5 Post-Conditions

* balance = balance - amount
* Insert transaction

### 8.6 Validation Priority

1. Auth
2. Schema
3. Account check
4. Balance check

### 8.7 Error Mapping

* INSUFFICIENT_FUNDS → 409
* ACCOUNT_NOT_FOUND → 404

### 8.8 Concurrency

* Strict lock required

---

## 9. Global Cross Field Rules

* IF status != ACTIVE → block transactions
* IF amount <= 0 → reject
* IF withdraw AND balance < amount → reject

## 12. Canonical Request & Response Payloads (All Endpoints)

All payloads strictly follow JSON with content-type `application/json`.

### 12.1 Common Success Envelope (if wrapped)

Field: success

* Type: Bool
* Required: True

Field: data

* Type: Object (expanded per endpoint below)
* Required: True

Field: request_id

* Type: String
* Min_Length: 1
* Max_Length: 64
* Required: True

Example:
{
"success": true,
"data": { ... },
"request_id": "req_01HZY..."
}

### 12.2 Common Error Envelope

Field: success

* Type: Bool
* Value: False

Field: error

* Type: Object

  * code: String (Enum from Error_Code)
  * message: String (Min_Length: 3, Max_Length: 256)
  * field: String (optional, may be null)

Field: request_id

* Type: String

Example:
{
"success": false,
"error": {
"code": "INVALID_SCHEMA",
"message": "Field initialDeposit missing",
"field": "initialDeposit"
},
"request_id": "req_01HZY..."
}
